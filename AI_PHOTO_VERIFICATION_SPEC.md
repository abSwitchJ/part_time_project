# 礼仪模特照片核验模块 · v2 产品设计

> ⚠️ **范围声明:本文档是 v2 规划 / 未来愿景,不属于 MVP 范围。**
>
> 对应 PRD `part_time_wechat_mini_program_project_mvp.html` §3.8《礼仪模特照片核验》里已明确延期的 **"AI 自动相似度 / 真实性核验"**。
> 它回答的是:**当 v1 的实时拍照采集 + 人工核验跑通、样本积累到位之后,AI 该以什么产品形态介入"礼仪模特照片核验"**——
> 不是"现在就把大模型接进一个还没上线的小程序"。
> v1(MVP)的照片核验仍然全部由中介方 / 平台**人工肉眼比对**完成,本文档描述的任何 AI 能力都**不在 v1 交付物里**。
> MVP 已交付的部分(实时核验照采集、15 天有效期、招聘方可要求重拍、人工核验)见 §3,**那部分不带 v2 标记**。

---

## 1. 定位与边界

**这是什么(v2)**:在应聘方提交礼仪模特 / 形象类报名(主流程 #14)之后、进入报名池被中介方人工筛选之前,平台用多模态模型对**实时核验照**与**模卡照**做一次自动比对,产出两个量化信号——人脸相似度、模卡照真实性 / 篡改可疑度——并给中介 / 平台一个**处置建议**:高置信真实者自动放行进池,其余转人工复核。

**这不是什么**:

- AI **不判"真假"**。"是否同一人"(人脸比对)、"模卡照是否被篡改"(P 图 / AIGC / 翻拍检测)、"真实形象与岗位的差距"(主观审美)是三件不同的事。人脸比对抓不到"同一个人的过度美颜照"——同一个人,照片仍可能与本人差距极大。所以 AI 的职责是**量化差距 + 标记可疑**,把一个**素颜对照基准**和一组分数交给人,不是替人下"这张是假的"的结论。
- AI **绝不自动拒绝**。处置建议只有二值:「自动放行」或「转人工复核」,**没有「自动拒绝」**(理由见 §2、§7)。
- AI **不改 §5 状态机、不替代人工录用决策**。它只是报名进池前的一个增强步骤;报名 / 筛选 / 双重确认(#17 / #18 / #19a)的语义与状态迁移完全不变。
- AI **不替代 MVP 的实时拍照采集**。实时拍照(防盗图 / 旧照)是 MVP 已实现的真实功能,AI 只是 v2 在它之上叠加的自动比对层。

一句话:**实时拍照采集真照片(MVP,人做);AI 量化差距、标记可疑、放行高置信(v2);录用决策永远是人做。**

## 2. 为什么 MVP 用人工、v2 才上 AI(核心产品判断)

这一节是本文档最重要的部分——它解释的不是"AI 怎么做",而是**"为什么现阶段照片核验不该交给 AI"**。

1. **两类错误的成本严重不对称,而冷启动期的模型必然两头都错。**
   - **漏判**(P 图照过关):到岗才发现人货不符,直接击穿招聘方对平台的信任——这正是这个功能要解决的核心痛点,漏判等于功能失效。
   - **误拒**(真人被判为造假):不只是丢一个用户,更糟的是"算法以貌取人""AI 说我长得假"会演变成舆情与伦理事件。礼仪模特是高度依赖外貌评价的人群,被算法判定"形象不真实"对个人是冒犯性的。
   - 两类错误一类伤业务、一类伤人且伤品牌,成本不在一个量级。结论很硬:**AI 只能对"高置信真实"自动放行,其余一律转人工,绝不自动拒**。这与 PRD §3.4「故意不替招聘方做默认通过」是同一种叙事——在代价不对称的地方,系统主动**不**做那个看起来更省事的决策。

2. **冷启动期没有"P 图 vs 真实"的标注样本,模型在猜。** v1 上线初期,平台手里没有"哪些模卡被 P 过、P 到什么程度、与本人差多少"的标注数据。任何篡改检测 / 相似度模型在没有这类样本时都是高方差的。正确顺序是:**先用 MVP 的人工核验跑通、并把每一次"人工判定可疑 / 通过"沉淀成标注**,再上模型。礼仪模特核验场景天然产出这种强标注——这是"先人工后 AI"路线的数据基础。

3. **MVP 的实时拍照已经挡住了主要造假手段,人工肉眼比对足够跑通。** 造假的大头是"盗用他人照片""用几年前的旧照""直接传一张精修图当生活照"。MVP 要求**在小程序内实时拍摄、禁止相册上传**,这一条就把盗图和旧照基本堵死;剩下的"本人但过度美颜"的差距,人工把实时照和模卡照并排一看就能判断。也就是说:**MVP 不上 AI 也能把功能跑通**,AI 是把人工核验提速、并在规模化后兜住人工看不过来的部分,不是闭环必需品。

> **面试用法**:这一节可以直接回答"你这个 AI PM 作品里,为什么照片核验这种'明显该上 AI'的场景你反而不上 AI"。答案不是"我没做",而是:"我判断现阶段不该把它交给 AI——因为误拒一个真人模特的成本(舆情 + 伦理)远高于漏判,而冷启动期的模型两头都会错;我先用实时拍照堵住主要造假手段、用人工核验攒标注,并且明确说得出 AI 该在什么数据条件下、以什么形态(只放行不自动拒)介入。"能讲清楚"AI 在哪不该用、为什么、什么时候才该用",本身就是 AI 产品判断力。

## 3. MVP 形态(v1,已在交付范围)

> 本节描述的是**已在 MVP 交付范围内**的功能,**不带 v2 标记**。

礼仪模特 / 形象类应聘的照片真实性,MVP 阶段用一套**纯产品机制 + 人工核验**解决,不依赖任何模型:

- **模卡新增「实时核验照」槽位**:该槽位只能通过小程序**调起相机实时拍摄**,**禁止从相册上传**。这是整个机制的地基——它把"上传一张图"这个可造假动作,换成"此刻、此设备、拍此人"这个难造假动作。
- **核验照有效期 15 天**:核验照拍摄后 15 天内视为有效。应聘礼仪模特 / 形象类目时:
  - 核验照**在有效期内** → 直接复用,无需重拍;
  - 核验照**过期 / 首次报名** → 必须当场实时拍摄才能完成报名。
- **招聘方可要求本次实时拍摄**:招聘方发布礼仪模特 / 形象类招募令时,可勾选「**要求本次实时拍摄核验照**」。勾选后,该单报名**无视 15 天有效期、强制重拍**。
- **照片随报名进入人工核验**:实时核验照连同模卡照一起进入中介 / 平台的**人工核验队列**,由人肉眼并排比对实时照与模卡照,判断是否一致;有疑义可在筛选环节标注。

**为什么 15 天有效期、而不是"每次报名都重拍"**:礼仪模特是典型的"一人多投"人群,一周可能报十几个单。每次报名都强制重拍,会在报名漏斗最前端加一道明显摩擦,直接压低报名转化率。15 天有效期 + 招聘方可按单强制重拍,把"采集摩擦 vs 真实性强度"做成一个**显式的、可调的产品决策**:平台默认偏向低摩擦,把"我这个单必须要最新照"的决定权交还给招聘方。

## 4. v2 介入点

- **位置**:应聘方提交礼仪模特 / 形象类报名(主流程 #14)**之后**、该报名进入报名池被中介方人工筛选**之前**。
- **流程**:应聘方实时拍摄核验照 → 提交报名(#14)→ **平台 AI 对「实时核验照 ↔ 模卡照」做相似度 + 真实性比对** → 高置信真实 → 自动放行进报名池;其余 → 标记可疑、转中介 / 平台人工复核 → 人工判定后进池。
- **明确不新增带编号的主流程消息**:**不**新增 `#x` 到主流程时序图。原因与 `AI_MATCHING_SPEC.md` §3 一致——PRD 的消息编号(#1–#22, #19a)被附录 A 三向索引承载性引用,且时序图有内嵌 SVG + 外置 `diagram-*.svg` 双份;新增编号成本高、收益低。照片核验是 #14 报名进池**之间**的一个增强步骤,对外可见的消息序列不变。
- **与 MVP 的关系**:v2 不替换 §3 的任何机制——实时拍照采集、15 天有效期、招聘方可要求重拍照旧运行;AI 只是**自动跑一遍人工本来要做的肉眼比对**,把高置信的放掉、把可疑的连同分数推给人。人工核验从"全量肉眼比"变成"复核 AI 标出来的那部分"。

## 5. 输入特征

| 类别 | 字段(示例) |
|---|---|
| 实时核验照 | 小程序内实时拍摄的当前生活照,带**拍摄时间戳 / 设备指纹 / 不可去除水印**;原则上素颜或接近素颜 |
| 模卡照 | 应聘方模卡中的 1~N 张展示照(含可能被 P 过的精修图) |
| 岗位形象要求 | 招募令中的形象类硬 / 软要求(身高、形象 tag、是否要求特定造型)——用于"差距"维度的上下文,不进"是否同一人"的判断 |
| 采集上下文 | 是否走"招聘方要求本次重拍"路径、核验照距拍摄的天数、是否首次采集 |

水印 / 时间戳 / 设备指纹是输入里最关键的一组:它们让"实时照"本身可被验真,模型不必只靠像素去赌这张照片是不是当场拍的。

## 6. 输出形态

每次核验输出一组结构化结果:

- `face_similarity`:0–1 连续值,实时照与模卡照的人脸相似度(回答"大概率是不是同一个人")。
- `tamper_score`:0–1 连续值,模卡照的篡改 / 美颜 / AIGC / 翻拍可疑度(回答"模卡照被加工得有多重")。
- `confidence`:置信度 bucket(高 / 中 / 低),反映模型对上面两个分数有多确定。
- `suggestion`:处置建议,**仅二值——「自动放行」或「转人工复核」,无「自动拒绝」**。
- `reasons`:2–3 条人类可读说明 tag,给人工复核者看。

示例:

```json
{
  "applicant_id": "u_20571",
  "recruitment_id": "r_8842",
  "face_similarity": 0.91,
  "tamper_score": 0.68,
  "confidence": "中",
  "suggestion": "转人工复核",
  "reasons": ["人脸高度相似,大概率同一人", "模卡照检出强美颜/瘦脸痕迹", "实时照与模卡照肤质差异明显"]
}
```

注意这个例子:人脸相似度很高(同一个人),但篡改分也高(模卡照被重度精修)。**这恰恰是 AI 不能自动拒、也不能自动放行的典型情况**——它不是"盗图",但"人货差距"真实存在,该不该用要人结合岗位来判。`reasons` 是这个模块能不能落地的关键:人工复核者只有看得懂"AI 为什么把这张标出来",才会信任并采纳建议;否则会直接忽略 AI、退回全量肉眼比对,模块白做。

---

## 7. 置信度处理与兜底

- **高置信 + 真实**(相似度高、篡改分低、confidence 高):`suggestion = 自动放行`,直接进报名池,不占用人工。
- **其余所有情况**(相似度存疑 / 篡改分偏高 / confidence 非高):`suggestion = 转人工复核`,连同分数与 `reasons` 进人工队列。**模型"没把握" ≠ "这个人造假"**——没把握就交给人,绝不让模型在没把握时下拒绝结论。
- **模型服务失败 / 超时**:**静默全量降级为人工核验**,该批报名照旧进人工队列,应聘方端无感、报名流程不阻塞、不报错。AI 不可用绝不能卡住 MVP 已经跑通的"实时拍照 + 人工核验"流程。
- **为什么没有"自动拒绝"档**:见 §2 第 1 条。误拒一个真人模特的成本(用户流失 + "算法以貌取人"舆情 + 伦理风险)远高于把可疑样本转人工的成本。把"拒绝"这个动作完全收回给人,是这个模块的头号设计约束,不是可调参数。

## 8. 冷启动与模型演进

> 这一节体现"按数据成熟度选模型、不一上来就 LLM"的判断。

- **v2.0 · 传统人脸比对 + 规则化真实性信号**(数据门槛最低):
  - 用成熟的人脸比对模型出 `face_similarity`;真实性维度先用规则信号——水印 / 时间戳 / 设备指纹是否完整、EXIF 是否异常、是否检出已知美颜 App 的处理痕迹。
  - 目的:在没有篡改标注样本时也能给出可用的"放行 / 转人工"分流,同时**产出第一批"AI 判定 vs 人工最终判定"的对照样本**。
- **v2.1 · 接入图像篡改 / AIGC 检测模型**(累积人工核验标注后):
  - 把规则化真实性信号升级为专门的篡改 / AIGC / 翻拍检测模型,`tamper_score` 从规则分变成模型分。
  - 此阶段才有足够的"P 图 vs 真实"标注来训 / 调这类模型。
- **v2.2 · 多模态大模型综合判断 + 自然语言核验理由**:
  - 用多模态模型综合相似度、篡改信号、岗位形象要求,生成更准的处置建议,并把 `reasons` 从 tag 升级为自然语言解释("模卡照在下颌线和肤质上有明显加工,但五官结构与实时照一致")。

**为什么不一上来就用多模态大模型**:① 冷启动期可解释性 > 拟合能力,人工复核者要看得懂理由,规则信号和传统模型的理由比大模型的"黑箱理由"更可信、更可审计;② 大模型推理成本高,而 v2.0 的分流大部分靠"水印 / 时间戳是否完整"这类低成本信号就能做掉;③ 大模型的真正增量价值在 `reasons` 的自然语言生成和综合判断,那是 v2.2 的增量,不是核验主引擎。

## 9. 隐私与合规(first-class)

人脸是《个人信息保护法》明确的**敏感个人信息**,本模块的合规约束是**设计约束级**的,不是脚注:

- **单独同意**:采集实时核验照、用于 AI 比对,必须有**独立于通用隐私政策的单独勾选同意**,告知用途、处理方式、存储期限。
- **最小必要**:只采集核验所必需的实时照,不顺带采集与核验无关的生物特征;比对完成即不再保留原始高清图。
- **明确存储期限 + 默认销毁**:实时核验照在**完成核验后默认销毁**;确需短期留存(如配合招聘方复核)的,留存期限明确、到期自动删除,且不超过核验业务所必需。
- **入训练集需再次授权**:核验照若要进入模型训练集,**必须再次取得单独授权**——"同意用于本次核验"不等于"同意用于训练 AI"。默认不入训练集。
- **未成年人处理**:礼仪模特群体可能含未成年人。未成年人的人脸采集需取得监护人同意;无法取得的,该类目对未成年应聘方走纯人工、不进 AI 比对。
- **可解释与可申诉**:被转人工 / 被人工判定不通过的应聘方,有权知道大致原因并申诉(见 §12)。

## 10. 评估指标

| 指标 | 含义 | 目标方向 |
|---|---|---|
| 漏判率(P 图过关率) | 被 AI 自动放行、但事后(到岗反馈 / 抽检)证实为重度造假的比例 | 越低越好——漏判直接伤招聘方信任 |
| 误拒率 / 误转人工率 | 真实无问题的应聘方被 AI 标为可疑 / 转人工的比例 | 越低越好(护住体验与伦理风险,但永远优先于漏判被压低) |
| 人工复核命中率 | 转人工的样本里,人工最终判定确有问题的比例 | 越高越好——太低说明 AI 把太多干净样本推给了人 |
| 人工核验耗时下降 | 接入 AI 后,单笔报名照片核验的人工平均耗时变化 | 显著下降(AI 的核心效率收益) |

**挂钩 `METRICS.md`**:

- **核心收益指标 = 到岗"人货不符"投诉率下降**——这是这个功能存在的理由,照片核验做得对不对,最终看招聘方到现场是否还在抱怨"人和模卡对不上"。
- **护栏指标 = 礼仪模特 / 形象类报名转化率**——实时拍照本身是一道摩擦,15 天有效期是为了压住这道摩擦。如果报名转化率被照片核验明显砸低,说明摩擦设计失败,收益是"借来的"。

## 11. AB 实验设计

- **分流单元**:应聘报名(单笔礼仪模特 / 形象类报名)。
- **对照组**:纯人工核验(v1 现状——实时照 + 模卡照全量人工肉眼比对)。
- **实验组**:AI 初筛(高置信真实自动放行)+ 人工复核(其余转人工)。
- **样本量**:目标 200 笔 / 组以上。
- **主指标**:人工核验耗时(预期下降)、人工复核命中率(预期上升)。
- **护栏指标**:漏判率不能高于纯人工基线(AI 提速不能以放过造假为代价)、报名转化率不能跌(AI 介入不应引入新摩擦)。

## 12. 失败模式与可解释性

| 风险 | 产品策略 |
|---|---|
| 误拒 / 误转人工伤体验、引发"算法以貌取人"舆情与伦理风险 | AI 永不自动拒(§7);被人工判定不通过者提供**申诉通道**,由人复核;申诉结果回流为标注 |
| 长尾:同一人因妆造 / 发型 / 光线差异被判低相似度 | 妆造差异属正常,不进自动拒;低相似度一律转人工,由人结合多张模卡照判断;长尾样本优先纳入后续训练 |
| 对抗:打印照翻拍、deepfake、对抗样本绕过检测 | 实时拍照 + 水印 / 时间戳 / 设备指纹做第一道防线;v2.1 篡改 / AIGC 检测做第二道;高对抗样本主动转人工,不赌模型 |
| 模型在特定人群(肤色 / 年龄 / 性别)上系统性偏差 | 分人群监控误拒率,发现系统性偏差即对该人群收紧"自动放行"阈值、扩大转人工范围,并复查训练样本分布 |
