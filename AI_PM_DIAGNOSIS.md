# 作品诊断:作为 AI PM 实习作品集的现状评估

**评估对象**:本仓库 `part_time_project`(PRD 1.3k 行 + 中介选模 React 原型 4 变体)
**评估视角**:作者是大三学生,正在找 **AI 产品经理实习**——所以这份诊断不评估"产品本身做得好不好",而评估"**这份作品拿出去面试 AI PM 实习,能拿多少分,还差什么**"。
**结论先行**:目前作品的**产品基本功**已经在合格线以上,**但 AI 含量结构性偏低**——不动手补就直接投简历,大概率会被问到 30 秒内答不上"你这个 AI PM 作品里 AI 在哪"。这是必须先解的问题,具体怎么解见同目录 `AI_PM_ROADMAP.md`。

---

## 一、五个维度盘点

### 1. 产品深度 —— 强(超出大三学生平均水平)

**证据**:

- **§3 关键设计决策** 的"决策 — 取舍 — 实现位置"三段式 callout(§3.1 双重确认 / §3.2 自动递补 / §3.3 统一取消 / §3.4 三段超时):每条都把"为什么不这样做"的反方案写出来了。这是 PM 训练有素的标志——大部分大三作品只写"我决定怎么做",不写"我为什么没那样做"。
- **§6 异常路径分类法**:把异常拆成"有状态、需要画时序图的 4 条"(§6.1)和"规则性强、表格刻画即可的 8 条"(§6.2),并在 §6 开头明说了分类标准。这是**方法论意识**,不是堆砌。
- **§2 v2 延期项的判断标准**:① 跨多系统 ② 不影响主流程 ③ 缺数据支撑——三条标准让"延期"看起来是产品决策而不是"做不完拖一拖"。
- **附录 B v3 vs v2 修订表**:6 处差异每处都有"v2 形态 / v3 形态 / 详见 §"三列——这本身就是一段**完整的产品迭代故事**,面试时可以说"我接手 v2 评审稿,识别出 6 处问题并修订,逻辑链如下"。

**弱项**:

- **PRD 本身几乎没有数据/指标支撑**(全文 grep 找不到目标成单率、双重确认通过率、递补成功率)。这是 §2 自己也承认的。**2026-05-18 更新**:这条短板已由 P0-2 `METRICS.md` 单独补齐——每条 §3.1–§3.4 决策配验证指标 + 假设阈值 + §4 证伪逻辑("什么数据涨/跌就推翻该决策"),北极星=双确认成单率。**残余诚实 caveat**:产品未上线,`METRICS.md` 所有阈值标注"假设·待验证",讲的是"会用数据思考"而非"已有实测数据"。

**面试该讲**:§3.1–§3.4 的取舍叙事 + 附录 B 的迭代故事 + `METRICS.md` 的验证指标/证伪逻辑(2026-05-18:已可主动讲指标体系,原"面试别提指标体系"已失效)。**仍需诚实**:被追问实测数据时,说清"产品未上线,这是指标设计与假设阈值,不是数据报告"。

### 2. AI 含量 —— ⚠ 这是核心张力

**证据**(全文 grep "AI / 算法 / 匹配 / 模型 / rank" 的结果):

- 整个 MVP 把"筛选 + 排序"的决策权**显式交给了中介方人工**(§1 中介方角色定义、#17 提交筛选结果)。
- AI/算法在 MVP 里只出现在三个 **fallback 位置**:
  - T1 中介超时后的"系统自动初筛"(§3.4)——表述为"按报名时间 + 简历匹配度",一句话,无规格。
  - "匹配引擎"作为责任方挂在 §6.2 表格"无匹配兼职"(P1)、"临近截止 24h 无人报名"(P1,处理策略写"匹配阈值放宽至 70%")两条上——一句话,无规格。
  - §2 范围表把"多维匹配算法"明确**延期到 v2**,MVP 只用"单维类别 + 时段匹配"。

**评估**:

这其实是一个**有意识的、可辩护的产品决策**——冷启动期用人工跑通业务、避免 AI 假阳性破坏四方信任、用人工生成的样本反哺训练集。"为什么 MVP 不上 AI 匹配"这道题如果答得好,**反而是 AI PM 面试的加分项**(体现"AI 不是万能、知道什么时候不该用 AI"的边界感)。

但作品**纸面上的张力**仍然是:

- 简历投出去之后,招聘方先扫的是"AI / 模型 / 算法 / 推荐 / RAG / Agent"这些关键词,目前作品里这些词的出现密度**极低**。
- §3 现有 4 条设计决策都没有一条是关于 AI 的——意味着用户的产品思考还没显式渗透到"AI 介入哪里、不介入哪里"这个维度。
- "中介选模"原型 (`model_selection_business/`) 本质上是 AI 推荐排序的天然落点(候选人 ranking + 解释)——但原型目前画的是**纯人工筛选界面**,没有 AI 辅助元素(置信度、推荐理由、相似案例)。

**面试该讲**:"为什么 MVP 不让 AI 替代中介人工筛选"——已落到 PRD §3.5(三段式 callout,标 v2·本版不做)。**2026-05-18 更新**:"AI 在你的产品规格里具体怎么落地"原本会哑火,现已由 `AI_MATCHING_SPEC.md` 补齐(介入点 / 输入特征 / 输出形态 / 置信度 / 兜底 / 冷启动规则→GBDT→向量召回演进 / 评估指标 / AB 设计 / 失败模式)。**残余 caveat**:这是 v2 设计预案,MVP 不实现——讲的时候要把"为什么 v1 不做"和"v2 怎么做"绑在一起讲,别让对方误以为 MVP 已有 AI。

### 3. 设计完整度 —— 中(三端关键路径已覆盖,仍是静态原型)

> **2026-05-18 更新**:本节原评级 ★★、原结论"只覆盖 1/4 视角"已过期。P1-2 扩展后,`part_time_job_wechat_mini_program_prototype/full-flow-prototype.html` 已把应聘 / 招聘 / 中介三端关键决策节点串成端到端一条流。评级上调至 **★★★**,但**封顶在 ★★★**——见下方"仍然薄"两条,不要对外说成"完整四方产品"。

**证据**:

- `part_time_job_wechat_mini_program_prototype/full-flow-prototype.html`:应聘方(#11 推送 → #14 报名 → #19a 确认接受 → 成单/完工)、招聘方(#9 发布 → #18 确认名单 → T11 完工确认)、中介方(报名池 → #17 提交有序 rank)三端关键路径都有可点击屏。
- `model_selection_business/` 三个视觉变体 + 一个 standalone bundle:「中介选模」子功能的多视觉风格探索(招募令列表 → 报名者列表 → 模特档案)。
- **仍然薄**:① 原型是**静态演示**,没有跨端实时状态同步(应聘方点"接受"不会真的回写中介报名池)——被追问"数据怎么从 C 端同步到中介端"只能答"原型是硬编码,技术实现留待真开发";② **平台运营后台**(审核 / 客服 / 数据)仍接近零原型。

**评估**:

最大遗憾**#19a 应聘方确认接受**已补上原型(v3 双重确认闭环的点睛动作,§3.1 callout 最核心的一笔),"用户看到这条通知时我怎么设计 24h 显式确认窗口"现在可现场打开演示。设计完整度的真实瓶颈已从"只画一方"转为"静态、缺平台侧"。

**面试该讲**:打开 full-flow 走一遍三端关键路径,落在 §3.1/§3.2/§3.4 的设计决策上;中介选模三视觉变体表达"设计探索意识"。**先发制人**:主动说"这是展示设计决策如何落地的交互原型,不是可用应用,数据流是硬编码的"——把静态这件事自己讲掉,而不是被问出来。**仍可能翻车**:被追问"平台运营后台长什么样"——答不上,如实说"还没画"。

### 4. 商业 sense —— 零(完全没碰)

**证据**(全文 grep,以及人工逐节扫描):

- **收费方 / 收费模型**:零提及。中介方对招聘方收服务费?平台对中介抽佣?应聘方有没有保证金?——PRD 一字未答。
- **目标用户画像**:只有一句"MVP 期间以学生为主"(§1),无访谈、无地域 / 学历 / 兼职偏好的细分。
- **竞品**:零提及青团社 / 兼职猫 / BOSS 直聘兼职板块。
- **市场规模 / TAM**:零。
- **GTM 策略**:零(没说怎么冷启动、靠什么渠道获客)。

**评估**:

PRD 不写商业是合理的(PRD 是规格不是 BP),但作品集**完全没有一份单独的商业 / 市场分析文档**就有问题——AI PM 实习面试很常问"这个产品的商业模式是什么 / 假如你是 CEO,你怎么做大它"。

**面试该讲**:没有,所以最好补一份。**面试翻车点**:"你这个项目预计的 take rate / GMV 是多少"——目前答不出。

### 5. 可讲性 / 简历卖点 —— 三个强故事 + 一个空洞

**三个能直接拿出去讲的故事**(每个都是"识别问题 → 提出取舍 → 量化收益"的标准 PM 叙事):

1. **v3 双重确认闭环**(§3.1 + 附录 B 第一行):"v2 里招聘方点完确认就直接拉群成单,等于把'应聘方被通知'等价于'应聘方已同意'。这会让两类失败被压到拉群之后才暴露:① 应聘方默认拒绝;② 应聘方根本没看到通知。我新增了'待应聘方确认'状态 + T3 24h 显式确认窗口,把这两类失败拦在拉群之前。"
2. **自动递补**(§3.2 + 附录 B 第三行):"v2 每次有人放弃都要走'通知中介 → 中介 24h 内补人 → 招聘方审批',一笔订单可能要 3-4 天。我让中介方在 #17 一次性提交 rank 1~N 入选 + N+1~M 备选的有序列表,递补在平台内自动闭环——成单时长从 3-4 天压到 1 天内,代价是中介在 #17 时多干一点活,反过来也提高了筛选质量。"
3. **三段超时的'故意不对称'**(§3.4):"T1/T3 系统替用户做决策(自动初筛 / 默认放弃),T2 故意不替招聘方做'默认通过'——因为 T1、T3 侧'无响应'等价于'无能力提供'或'放弃',T2 侧'无响应'不能替代招聘方对名单的判断。这条不对称是有意的——这是我对'系统什么时候该替用户决策'的边界判断。"

**半个空洞**(2026-05-18:从"整个空洞"降级):

- 第四个故事的**素材已具备**——`AI_MATCHING_SPEC.md` 写清了"为什么 v1 不做 / v2 怎么做 / 按数据成熟度从规则→GBDT→向量召回演进",这正是能撑 5 分钟深入追问的金矿。
- **仍空的那半**:素材**尚未提炼成一段可口述的面试故事**。`INTERVIEW_NOTES.md`(路线图反复提醒要建的面试自答稿)**仍未建**——"做出来了"不等于"临场讲得出"。这是当前可讲性的真实缺口。

## 三、面试 Talking Points 备忘

**主动讲(按顺序)**:

1. "我从 v2 接手,识别了 6 处问题做了 v3 修订,核心是双重确认闭环"——开场亮活儿,拉到 §3.1 + 附录 B
2. "我没让 AI 替代中介人工筛选,因为……"——主动堵掉"AI 在哪"这个问题,把"不用 AI"包装成产品判断(**前提是 §3.5 写出来**)
3. "T1/T3 系统替用户决策,T2 故意不替——这是我对边界的判断"——展示边界感
4. "我把成单时长从 3-4 天压到 1 天内,代价是中介方多干一点活"——展示量化收益思维

**被动应对**:

- Q: "AI 具体怎么落地?" → 拉到 `AI_MATCHING_SPEC.md` ✅ **已交付(P0-1,定位 v2 愿景)**
- Q: "你的北极星指标是什么?" → 拉到 `METRICS.md` ✅ **已交付(P0-2,北极星=双确认成单率;阈值均为假设·待验证)**
- Q: "C 端确认接受界面是什么样的?" → 拉到 `part_time_job_wechat_mini_program_prototype/full-flow-prototype.html` ✅ **已画(#19a 在内)**
- Q: "竞品怎么看?" → `COMPETITIVE_ANALYSIS.md` ❌ **尚未补(P1-1)**,正式海投前补;现在被问只能如实说"竞品分析在整理中"
