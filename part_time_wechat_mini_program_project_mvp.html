一、角色与核心术语

本产品有 4 个参与方。所有图表、消息编号、状态迁移都围绕这 4 个角色展开。读图时如果忘了谁是谁,翻回这一节。

应聘方(C端用户)
浏览招募令、报名、应答入选通知。MVP 期间以学生为主。关键动作:#14 报名、#19a 确认接受(v3 新增显式确认)。
兼职小程序(平台)
业务编排中枢——状态机、超时调度、消息推送、群聊建立都由平台承担。关键动作:所有 T1–T5/T11 超时触发、状态迁移、自动递补激活。
招聘方(B端商户)
发布招募令、确认入选名单、确认完工。关键动作:#9 发布、#18 后确认名单(v3 拆出来的双重确认上半段)。
中介方(平台预入驻方)
代招聘方做候选人初筛 + 维护备选池排序。MVP 期间为平台预先签约方,不开放第三方入驻。关键动作:#17 提交筛选结果(同时给出 rank 1~N 入选 + N+1~M 备选)。

核心实体:招募令(Recruitment Order)

产品的核心业务实体。从招聘方点击"新建"开始,到"已关闭/已过期/已取消"结束,生命周期会经历约 10 个状态(见 §5)。一条招募令绑定:1 个招聘方、1 个中介方、N 个入选应聘方、M 个备选应聘方、1 个沟通群聊。
工程视角:对应数据库 recruitment 表,status 字段枚举值见 §5.3。

核心机制:双重确认闭环 ★

v3 的核心交付物。一笔订单要成立,需要招聘方确认名单(#18)和应聘方确认接受(#19a)两步显式确认,任一阶段中断都进入对应的恢复流程(异议重筛 / 自动递补)。详见 §3.1。

端架构:一个小程序 · 多种身份

决策:四方共用同一个微信小程序,登录后按账号身份(应聘方 / 招聘方 / 中介方)分流到不同主界面,而非拆成 C 端小程序 + B 端小程序 + 中介后台。
取舍:主流招聘类产品多走 BOSS 直聘 / 美团商家版那条路——C 端、B 端各自独立 App。这条路的好处是 B 端操作复杂度天花板高(可接 PC 后台),坏处是冷启动期获客成本翻倍、品牌认知分散。MVP 阶段做反向选择有三条理由:① 招聘方多为小商户(街边奶茶店店长、社区便利店主),不会专门开 PC 后台,微信小程序就是其唯一接触面;② 中介方为平台预入驻方,操作量级在小程序内可承载,真上规模再迁后台;③ 应聘方=学生,习惯微信生态、避免再装 App。代价是 B 端操作复杂度被压缩到小程序极限,招聘方批量管理类需求(同时跟踪 N 条招募令)推到 v2 工作台(见 §2 延期项「多招募令并发管理工作台」)。
实现位置:登录后按 user.role 分流到三种主界面(应聘方 = 职位 / 日程 / 我的;招聘方 = 招聘 / 我的;中介方 = 首页 / 待办 / 我的)——见原型 part_time_job_wechat_mini_program_prototype/full-flow-prototype.html 的 TABS 实现;身份的首次选定见该原型顶部「进入 · 选择身份」屏(三张角色卡:应聘方 / 招聘方 / 中介方,选定后即按上述 user.role 分流到对应主界面)。状态机(§5)对身份无感,所有角色操作都走同一套招募令生命周期。

二、MVP 范围声明

MVP 的目标:跑通一条完整的"发布 → 报名 → 筛选 → 双方确认 → 成单 → 完工"主流程,并能稳定处理 4 类规则化异常(审核驳回 / 中介超时 / 招聘方异议 / 应聘方放弃)。其他诉求一律走 v2。
MVP 范围内延期到 v2
四方完整主流程(#1–#22) 多招募令并发管理工作台
招募令 10 个状态 + 3 个终态 履约期违约的分级补偿策略
候选人子状态机(SELECTED/BACKUP → PENDING → CONFIRMED/DECLINED/ABORTED/SKIPPED) 按取消时点分级处理退款(发布后取消 vs 成单后取消)
双重确认闭环(招聘方 #18 + 应聘方 #19a) 第三方中介开放入驻(MVP 期间中介方为平台预先签约方)
自动递补(按中介方提交的 rank 顺序激活备选) 多维匹配算法(MVP 期间使用单维类别 + 时段匹配)
礼仪模特 / 形象类应聘的实时核验照采集 + 人工照片核验(见 §3.8) AI 自动照片相似度 / 真实性核验(见 §3.8、AI_PHOTO_VERIFICATION_SPEC.md)
4 类异常路径(见 §6) 群聊解散后历史消息归档
T1(中介筛选 24h)/ T2(招聘方确认 48h)/ T3(应聘方应答 24h)/ T4(变更应答 24h)/ T5(截止日期)/ T11(完工确认 7d) 异常自愈机制(系统检测到状态异常自动修复)
统一取消处理:直接置 CANCELLED + 通知所有相关方 + 解散群 取消时差异化补偿、财务对账接口、客服申诉工单

三、关键设计决策(本版闭环要点)

这一节回答"为什么这么设计"。每条决策都有一个明确的取舍——读完这一节,看后面的图表就不会有"为什么不这样"的疑问。

3.1 双重确认闭环:为什么把"成单"拆成两段?

决策:在"待招聘方确认"和"已成单"之间新增 待应聘方确认 状态,应聘方走 24h 显式确认窗口,默认放弃。
取舍:v2 版本里招聘方点完确认就直接拉群、置已成单,事实上把应聘方"被通知"等同于"已同意"。这会导致两类失败被压到拉群之后才暴露:① 应聘方默认拒绝;② 应聘方根本没看到通知。这些失败到了"已成单"再处理,无论是数据账、群聊状态、还是招聘方信任度都会出问题。
实现位置:主流程 #19a(§4)、状态机 PENDING_APPLICANT(§5.3)、候选人子状态机 PENDING(§5.2)、T3 超时(§7)。

3.2 自动递补:为什么递补不再回中介方?

决策:应聘方放弃/超时后,平台按中介方提交的 rank 顺序自动激活下一备选,不重新走"通知中介 → 中介筛选 → 招聘方审批"。只有备选池耗尽才退回中介方。
取舍:v2 版本里每次有人放弃,都要走一遍"通知中介 → 中介在 24h 内补人 → 招聘方审批",一笔订单可能要 3-4 天才能成单。本版让中介方在 #17 提交筛选结果时一次性给出 rank 1~N 入选 + rank N+1~ 备选的有序列表,递补完全在平台内闭环。这把递补路径从"3 方协作"压成了"平台内自动",平均成单时长可压到 1 天以内。代价是中介方在 #17 时要多干一点活——但这一点反过来也提高了中介筛选的质量(因为他必须排出至少 N+M 个有序候选)。
实现位置:状态机递补中→待应聘方确认迁移(§5.1)、异常路径 alt 4(§6.1)、候选人子状态机 BACKUP → PENDING(§5.2)。

3.3 统一取消:为什么 MVP 不分级处理退款?

决策:招聘方在所有非终态(发布后到已成单前)主动取消,统一直接进入"已取消"——通知所有相关方 + 取消该单全部待办(含已建群聊解散),不处理退款 / 不区分补偿等级。
取舍:"按取消时点差异化补偿"是个看起来很合理的诉求(刚发布就取消 vs 应聘方已确认后取消,显然责任不同),但实现上要打通支付 / 财务对账 / 客服申诉三条链路,工程量 ≥ 主流程一半。MVP 阶段我们容忍"招聘方滥取消"的可能性,用统一通知 + 数据埋点先观察实际发生率;v2 再引入分级补偿。
实现位置:状态机多状态 → CANCELLED 迁移(§5.1)、异常处理表"招聘方主动取消"行(§6.2)。

3.4 三段关键超时:T1 / T2 / T3 各自防什么?

决策:主流程"筛选 → 双重确认"段引入三段超时,任何一段卡住都不会让招募令悬挂。
  • T1(24h,筛选中):防中介方在 #17 不响应——超时后系统自动初筛(按报名时间 + 简历匹配度),让招聘方仍能拿到候选名单。
  • T2(48h,待招聘方确认):防招聘方在 #18 后不响应——24h 提醒 + 48h 强制视为退回,触发异议重筛。不替招聘方做"默认通过"的决策。
  • T3(24h,待应聘方确认):防应聘方默认拒绝——视为放弃,触发递补(走 §3.2)。
原则:T1/T3 系统替用户做决策(自动初筛/默认放弃)是因为这两侧的"无响应"事实上等价于"放弃"或"无能力提供";T2 系统不替招聘方决策,因为招聘方对名单的判断不可由系统替代。这条不一致是有意的。
实现位置:§7 超时清单。
另:T2/T3 倒计时窗口内,中介方可向被等方(招聘方 / 应聘方)发软提示(催办)——不替对方决策,但允许加速响应,频控 24h 一次,详见 §3.7。

3.5 AI 辅助筛选:为什么 v1 不做、v2 才做? v2 · 本版不做

范围标记:本节是 v2 设计预案,MVP(本版 v3)不实现。列在 §3 是为了讲清"AI 该不该进、何时进"这个产品判断,与 §2 范围表、§8 v2 延期项(多维匹配算法)、§6.3 口径说明一致;v1 的筛选排序仍全部由中介方人工完成。
决策:v2 在"筛选中(SCREENING)"内部、#17 之前,由平台 AI 先把报名池按相关性排序 + 标出 top-K,中介方在排好序的池子上做最终 rank 决策与背书。AI 只做粗筛与排序建议,不做入选决策、不碰 §3.1 双重确认闭环、不改 §5 状态机。
取舍:理论上可让 AI 直接出最终名单、砍掉中介人工(成本最低);MVP 不这么做、且 v1 完全不上 AI,三条原因:① 冷启动无成单标注数据,模型必然高方差,AI 假阳性会击穿招聘方对名单的信任——招聘方一旦不信任,§3.1 招聘方确认(#18)通过率就崩,闭环失效;② v1 中介人工筛选过程本身在产出高质量标注样本(谁入选/谁备选/谁淘汰),这是双方撮合竞品拿不到的数据资产,正确顺序是先用人工攒数据再上模型;③ v1 不依赖 AI 也能跑通(§2 单维类别+时段够用),AI 是效率增益不是闭环必需。代价:v2 落地时中介方要从"从零翻池"转成"在 AI 排序上复核",故 AI 理由必须可解释,否则中介直接忽略、模块白做。
实现位置:完整产品设计(介入点 / 输入输出 / 冷启动到模型演进 / 评估指标 / AB / 失败模式)见仓库根目录 AI_MATCHING_SPEC.md;PRD 内对应 §6.3 匹配引擎口径说明 与 §8 v2 延期项。§3.4 的 T1 自动初筛是 v1 无 AI 时的兜底,v2 由本模块接管。

3.6 建群机制:从"平台抽象拉群"到"中介上传二维码 + 平台 fan-out 图片通告"

决策:v2 把 #20-#22 表达为"平台拉三方进群"——一个抽象动作,不说明"谁建、用什么形式、双方怎么收到"。MVP(v3)把这一段拆成三个可观测的离散事件:#20 平台通知中介建群(双方 #19a 确认后,在中介方"待办"出现"已成单·待建群"卡片)→ #21 中介上传微信群二维码(中介在微信内手工建群后,把群二维码图片上传至小程序)→ #22 平台以图片通告 fan-out 至应聘方+招聘方(双方在小程序内收到带图片的系统通告,长按二维码扫码进群)。
取舍:理论上"平台直接建群"语义最干净,#20-#22 一笔带过即可;但 v3 拆开这一段,三条原因:① 微信能力边界——微信小程序 OpenSDK 不开放"代用户建群"能力,平台无法绕过用户在微信生态内建群,这是工程现实而非产品偏好;② 承接 §3.4 中介职能定位——中介本就是平台预入驻的执行方(§1 角色定义),由中介承接"建群+上传二维码"语义自洽,不需要引入新角色或新外部服务;③ 可埋点的离散事件优于不可观测的状态变更——v2 一句"拉群"无法做埋点(平均建群耗时、上传成功率、双方扫码转化率全部缺失),v3 三个事件分别可埋,直接对应 METRICS.md §3.1 的"拉群后 24h 内退群率"等指标。代价:中介在 #17 提交后多一道"上传二维码"动作,平台需要支持"图片消息"类型(v2 系统消息均为纯文本/状态变更,见 §4 主流程 #11/#12/#15/#19)。
实现位置:主流程 §4 #20-#22 表格 + 时序图;中介方原型新增"已成单·待建群"待办卡 + PlatformGroupSetup 上传二维码屏(见 part_time_job_wechat_mini_program_prototype/full-flow-prototype.html);双方端接收图片通告卡(应聘方 ApplicantDeal + 招聘方 EmployerMine);异常处理见 §6.2 "群聊扩容失败" + "用户拒绝入群" 两条;附录 B 第 7 行 v3 修订。

3.7 催办机制:不替对方决策,但允许中介软推进

决策:T2(待招聘方确认)/ T3(待应聘方确认)倒计时窗口内,中介方在待办详情屏可点击"催办",平台向被等方发一条非状态变更的提醒消息——独立于 §4 主流程 #1-#22 编号,不挂号、不改状态机。同一招募令 24h 内最多催 1 次(防滥催)。
取舍:① 不与 §3.4「故意不替招聘方默认通过」冲突——催办是软提示(由人发起、由对方决策),而非系统替对方决策,§3.4 强决策边界完整保留;② 不进 §6.2 异常表——异常表是兜底通道(系统替人善后),催办是主动加速通道(人推系统),通道语义不同;③ 不挂主流程编号——挂 #X 会让 #20-#22 之后编号膨胀,而催办本质是非状态变更通告,概念上等同于运营推送,挂号会污染主流程的"状态变更骨架";④ 24h 频控是为防"无意义反复催办伤害招聘方/应聘方对平台的信任",不是工程兜底。代价:中介有了一个轻动作权限,§3.4 的"强决策不对称"叙事旁边多了"软推进对称"叙事——这两条叙事拼起来才完整:"是否替对方决策" 和 "是否允许中介加速对方" 是两个独立的产品判断。
实现位置:中介方原型新增 PlatformWaitDetail 屏(part_time_job_wechat_mini_program_prototype/full-flow-prototype.html),三态 kind(employer / applicant / dispute):前两态在底部 Primary 按钮"催办对方 →",点击切 Ghost 灰态"✓ 已催办 · 24h 内不可重复";dispute 态走"重新提交筛选 →"路径不显示催办。平台侧推送渠道复用 §4 #19 已有的"异步推送(站内 + 短信 + 公众号)",文案模板由平台维护(MVP 内 1 个模板覆盖 T2/T3 两路)。

3.8 礼仪模特照片核验:为什么 MVP 用实时拍照+人工比对、AI 自动核验留 v2?

背景:礼仪模特 / 形象类是本平台的重要兼职类目。痛点很直接——应聘方的模卡照普遍被 P,与本人差距大,招聘方常常到了现场才发现"人货不符",这是一条直接击穿招聘方信任的造假缺口。本节回答"照片真实性该怎么核、AI 该不该进、何时进"。
决策:模特 / 形象类应聘强制采集实时核验照——模卡新增「实时核验照」槽位,只能在小程序内实时拍摄、禁止相册上传;核验照有效期 15 天,有效期内复用、过期或首次报名必须当场拍;招聘方发布该类招募令时可勾选「要求本次实时拍摄核验照」,勾选则无视有效期强制重拍。实时核验照连同模卡照进入中介 / 平台的人工核验队列肉眼比对。在此基础上,由大模型自动比对实时照与模卡照的相似度 + 真实性、给出"放行 / 转人工"建议属 v2 · 本版不做,且 AI 绝不自动拒绝。
取舍:① 为何不每次报名都重拍——礼仪模特一人多投,每次都重拍会在报名漏斗最前端加一道明显摩擦、压低报名转化率;15 天有效期 + 招聘方可按单强制重拍,把"采集摩擦 vs 真实性强度"做成显式、可调的产品决策(平台默认低摩擦,把"我这单要最新照"的决定权交还招聘方)。② 为何不只信模卡——模卡照可被任意 P 图、可盗用、可用旧照;实时拍照把"上传一张图"换成"此刻、此设备、拍此人",一条就堵死盗图与旧照。③ 为何 MVP 不上 AI(三条):冷启动期没有"P 图 vs 真实"的标注样本,模型必然高方差;需先用 MVP 人工核验跑通并攒标注;实时拍照已挡住主要造假手段,人工把实时照与模卡照并排一看即可跑通,AI 不是闭环必需品。④ 为何 AI 不自动拒绝——漏判(P 图过关)伤招聘方信任,误拒(真人被判假)= 用户流失 + "算法以貌取人"舆情与伦理风险;两类错误成本严重不对称,故 AI 只对"高置信真实"自动放行、其余一律转人工,不自动拒。这与 §3.4「故意不替招聘方默认通过」是同一种"在代价不对称处主动不做省事决策"的叙事。
实现位置:完整 v2 产品设计(介入点 / 输入输出 / 冷启动到模型演进 / 隐私合规 / 评估指标 / AB / 失败模式)见仓库根目录 AI_PHOTO_VERIFICATION_SPEC.md;§2 范围表已新增"实时核验照采集 + 人工核验"(MVP)与"AI 自动照片核验"(v2)两行;原型「实时拍照核验」屏见 part_time_job_wechat_mini_program_prototype/full-flow-prototype.html(应聘方 browse → 礼仪模特岗位详情 → 实时拍照核验);异常处理见 §6.2"拒绝相机授权 / 拍照失败""AI 照片核验服务不可用"两行。模卡、实时核验照等术语就近在本节定义:模卡= 形象类应聘方的形象资料卡(comp card),实时核验照= 小程序内实时拍摄、用于与模卡照比对的当前生活照。

4.5.1 应聘方视角(5 步)

  1. 收到匹配推送 — #11 — 状态 已发布(PUBLISHED) — 「推送 · FEED · 为你推荐兼职」 屏。 卡片显示招聘方 / 类别 / 时段 / 报酬 / 规则匹配标识;点任一条进入岗位详情屏后报名。
  2. 提交报名意向 — #14 — 候选人子状态 已报名(APPLIED) — 「报名 · APPLY · 报名这次机会」 屏。 提交后落回报名池,等待 中介方筛选 → 招聘方确认 → 本人确认 三步;首位报名触发"已发布 → 筛选中"。
  3. 24h 显式确认接受(关键) — ★ #19a — 待应聘方确认 → 已成单(超时则进入自动递补) — 应聘方「我的」主页 → 入口卡 「录用通知 · 确认接受 · #19a · 待你 24h 内确认」 → 进入 「24h 倒计时确认屏」。 屏底两按钮 「确认接受」 / 「放弃这次机会」 + 倒计时 HH:MM:SS。 这是 §3.1 双重确认闭环的应聘方一半——明示决策,不假设默认通过;超时按 §3.4 T3 视为放弃,平台按中介 rank 顺序自动递补下一备选(§3.2)。
  4. 入群履约 — #22 — → 已成单(FULFILLED) — 「成单 · DEAL · 从确认到完工」 屏首段「群通告 · 二维码已收到」+「已成单 · 进入履约」标识。
  5. 打卡 / 完工 / 评价 — 关联 T11 7D 窗口 — 已成单 → 已关闭(CLOSED) — 「到岗 · CHECK-IN · 当天班次打卡」 屏点 「我已完工」 上报 → 招聘方在 「完工 · DONE · 确认完工」 屏验收 → 应聘方回 「成单 · DEAL」 屏发布星级 + 留言评价,整笔招募令闭环。

4.5.2 招聘方视角(3 步)

  1. 发布招募令 — #9 — 草稿 → 审核中 — 「发布 · POST · 发一条招募令」 屏。 填类别 / 时段 / 地点 / 报酬 / 需求人数(另留备选)等字段,提交平台审核(合规 / 资质),通过后对外可见。
  2. 48h 确认入选名单(关键) — ★ #18 — 待招聘方确认 → 待应聘方确认 — 招聘方「我的」主页 → 入口卡 「确认入选名单 · #18 · 中介已提交待你确认」 → 进入 「确认入选名单」 屏。 rank 1–2 标"入选" / rank 3–4 标"备选" + T2 倒计时 48H;两种操作:「全部接受 → 进入待应聘方确认」/「异议退回 → 转异议重筛(alt 3)」。 §3.4 T2 故意不替招聘方做"默认通过"——超时视为退回,不视为通过,转 异议重筛(DISPUTED)。
  3. 7d 完工确认 — T11 — 已成单 → 已关闭 — 「完工 · DONE · 确认完工」 屏。 确认后触发结算与互评(招聘方对入选应聘方逐张打分,可单张独立发布)。

4.5.3 中介方(平台侧)视角(4 步)

  1. 监看报名池 — 无独立消息编号 — 状态 筛选中(SCREENING) — 「报名池 · POOL · 排序并提交」 屏。 展示当前所有报名者头像 / 标签 / 中介评注,▲▼ 调序,RANK 标签实时更新。
  2. 提交有序 rank(关键) — #17 — 筛选中 → 待招聘方确认 — 「报名池 · POOL」 屏底 「提交筛选结果」 按钮。 一次性提交 rank 1~N 入选 + N+1~M 备选—— 这是 §3.2 自动递补能做到"平台内自动闭环"的源头:任何入选者放弃,平台直接按这个 rank 顺序激活下一备选,不再回中介。
  3. 建群 + 派二维码 — #20–#22 — → 已成单(FULFILLED) — 双方 #19a 都确认后,「待办 · TODO · 按截止排序」 屏出现「已成单 · 待建群」卡 → 进入 「建群 · GROUP · 上传二维码」 屏,上传后点 「发送给应聘方和招聘方」 触发 fan-out;微信代建群限制见 §3.6。
  4. alt 3 异议重筛 — 关联异常 alt 3 — → 异议重筛(DISPUTED) — 「待办 · TODO」 屏「alt 3 · 异议重筛」卡 → 进入 「等待详情」 屏,必须回报名名录重新筛选(不能直接重交上次结果);招聘方异议的候选在 「报名池 · POOL」 屏带红色「招聘方异议」角标。

5.2 候选人(子)状态机

"待应聘方确认"在招募令层面是一个状态,但内部实际上是 N 个候选人各自的状态机 的集合。中介方在 #17 提交筛选结果 时一次性给出 (rank 1~N 入选 + rank N+1~ 备选) 的有序列表,后续递补按 rank 顺序自动激活。当所有候选人状态收敛 (全部 CONFIRMED / DECLINED / 池耗尽),招募令再做整单结算 → 已成单 / 筛选中。

六、异常路径

异常分两类处理:

6.3 "匹配引擎"口径说明(MVP vs v2)

§6.2 表格中"无匹配兼职""临近截止 24h 无人报名"两条所写的「匹配引擎」,在不同版本下指的不是同一个东西,这里一次性说清,避免被当成黑盒:

  • MVP(本版):匹配引擎 = 规则化单维匹配(类别 + 时段),与 §2 范围声明一致。"无匹配"= 规则未命中,静默不推送、后续命中再推;"临近截止 24h"= 规则阈值放宽至 70%。MVP 不含任何模型/AI。
  • v2:匹配引擎升级为 AI 辅助筛选排序模块(对应 §8 延期项"多维匹配算法")。其完整产品设计(介入点、输入输出、冷启动演进、评估与 AB)见仓库根目录 AI_MATCHING_SPEC.md——该文档属 v2 愿景,不在本 MVP 交付范围内。

八、v2 延期事项

下列 7 项在评审时被明确延期到 v2,本 MVP 不实现。延期的判断标准:① 跨多个系统(支付/财务/客服);② 不影响主流程跑通;③ 现阶段没有数据支撑设计决策。

延期项理由分类v2 触发条件
取消差异化补偿跨支付 + 客服MVP 数据观察到取消率 > 10% 或客诉率显著
履约期违约分级补偿跨支付 + 客服有实际履约违约案例发生
第三方中介开放入驻不影响主流程MVP 跑通后启动招商
多维匹配算法无数据支撑累积 1 万+ 报名记录后
多招募令并发工作台不影响主流程招聘方人均同时管理 ≥ 3 单时
群聊历史消息归档不影响主流程用户主动反馈或合规要求
异常自愈机制无数据支撑异常人工介入率成为瓶颈时

其中「多维匹配算法」的 v2 产品设计(AI 辅助筛选排序:介入点、输入输出、冷启动到模型的演进路径、评估指标与 AB 实验)已展开在仓库根目录 AI_MATCHING_SPEC.md。该文档明确定位为 v2 愿景,不在本 MVP 交付范围内;v1 的筛选排序仍全部由中介方人工完成。

关于 MVP 取消处理的额外说明:本版统一处理("通知 + 解散群,不退款不分级")在评审时是有争议的,反对意见主要是"招聘方滥取消会伤害应聘方信任"。我们的应对:① 数据埋点跟踪取消行为(取消时点、取消频次、取消理由分布);② 客服处理通道仍开放,个别恶劣案例可人工介入;③ 一旦数据触线立即启动 v2 的分级补偿设计。
