# 指标体系 / 北极星指标

> **这份文档是什么:** 兼职小程序 MVP 的**指标设计与验证假设体系**——把
> PRD §3.1–§3.4 每条关键设计决策,显式翻译成"上线后哪个数涨/跌,能证明
> 这个决策对/错"。
>
> **它不是数据报告。** 产品尚未上线,本文档**没有任何实测数据**,所有阈值
> 均为**上线前假设**,标注为"假设·待数据验证"。AI PM 面试考的不是"你现在
> 有没有数据",而是"你会不会用数据思考"——即:你能不能把设计假设拆到
> 指标粒度、并说清楚什么结果会让你推翻自己的设计。本文档回答的就是这个。
>
> **怎么读:** §1 北极星指标(一句话回答"什么数能代表整盘健康")→ §2 每条
> 设计决策的验证指标(回答"§3.x 这个决策怎么验证")→ §3 护栏指标(回答
> "为了 §1 上涨,会不会有别的东西被牺牲")→ §4 证伪逻辑(回答"什么结果
> 出现,我就认为设计错了,然后怎么改")。口径中的状态名 / 消息编号 / 超时
> 编号与 PRD §3、§5、§7 一致,不另起一套。

---

## §1 北极星指标(North Star Metric)

**北极星 = 双确认成单率**

> **口径:** 统计窗口内,*状态曾到达「已成单」且其后 7 日内未转入
> 「已取消」* 的招募令数 ÷ 同窗口内*新发布的招募令总数*。
>
> 分子要求"7 日内未取消",而不是"到达已成单即计数"。

**为什么是它(而不是 GMV / 成单数):**

- 它**同时反映 §3.1 + §3.2 + §3.4 三条核心决策**的健康度:招募令要走到
  "已成单",必须穿过双重确认(§3.1)、必要时的自动递补(§3.2)、且没有
  在三段超时(§3.4)里悬挂。任一条决策设计失效,这个数会先跌。
- **它抗刷量。** "7 日内未取消"这个尾巴,让"先冲一个已成单再取消"无法
  做高指标——这正对应 §3.1 要解决的核心问题:v2 把"被通知"当成"已同意",
  失败被压到成单之后才暴露。北极星把这类"虚假成单"直接扣掉。
- 它是**业务结果指标**,不是过程指标:对外能讲"我用一个数衡量四方闭环
  是否真的闭合",对内能驱动 §2 各过程指标。

**假设健康区间:** 上线 3 个月内 ≥ 60%(假设·待验证)。低于此说明闭环
在某一段持续漏单,需下钻到 §2 定位是哪条决策出了问题。

## §2 设计决策 → 验证指标映射

每条决策配 1–2 个**验证指标**(主),而非堆指标。阈值列均为
**上线前假设**;"数据来源"指向 PRD 的状态迁移 / 超时触发,埋点字段级规格
留待 `TRACKING_SPEC.md`(路线图 P1-3)。

| 设计决策(PRD) | 验证指标(主) | 假设阈值(待验证) | 数据来源 |
|---|---|---|---|
| **§3.1 双重确认闭环** | 招聘方确认(#18)→ 应聘方确认(#19a)**双段通过率** = 两段均显式通过的招募令 ÷ 进入「待招聘方确认」的招募令 | ≥ 75% | T2 触发日志 + #18/#19a 应答事件(状态机 §5.1 / §5.3 PENDING_APPLICANT) |
| **§3.1 双重确认闭环** | 拉群后 24h 内退群率 | ≤ 5%(应显著低于 v2) | 群聊成员变更事件(#19→#22 段) |
| **§3.2 自动递补** | 平均成单时长(发布 #9 → 到达「已成单」) | ≤ 24h | 招募令状态迁移时间戳(§5.1) |
| **§3.2 自动递补** | 递补激活成功率 = 备选池**未耗尽前**完成成单的招募令 ÷ 触发过递补的招募令 | ≥ 60% | REPLACING 进出迁移日志 + 候选人子状态机 BACKUP→PENDING(§5.2) |
| **§3.3 统一取消** | 招聘方主动取消率(发布后到已成单前任一非终态)= 进入「已取消」且取消方为招聘方 ÷ 新发布招募令 | < 10% | CANCELLED 迁移事件 + 取消发起方(§5.1 / §6.2) |
| **§3.4 · T1**(中介筛选 24h) | T1 超时率 = T1 触发 ÷ 进入「筛选中」的招募令 | ≤ 15% | T1 触发日志(§7) |
| **§3.4 · T2**(招聘方确认 48h) | T2 超时后**异议重筛挽回率** = DISPUTED 后仍走到已成单 ÷ T2 触发数 | ≥ 50% | T2 触发日志 + DISPUTED 出向迁移(§5.1 / 异常 alt 3) |
| **§3.4 · T3**(应聘方应答 24h) | T3 超时率 = T3 触发 ÷ 进入「待应聘方确认」的候选人应答总数 | ≤ 20% | T3 触发日志 + 候选人 PENDING→放弃(§5.2 / 异常 alt 4) |

> §3.4 的三个指标刻意分开:T1/T3 是"系统替用户决策"的两侧,T2 是"系统
> **不**替招聘方决策"的一侧(§3.4 原则)。三个数要分别看——把它们汇总成
> 一个"超时率"会抹掉这条**有意的不对称**,等于丢掉 §3.4 最值得讲的设计点。

## §3 护栏 / 反指标(防止为抬北极星而牺牲别处)

北极星上涨不一定是好事——以下三条若恶化,说明涨幅是"借来的":

1. **应聘方端 NPS 不能跌。** §3.1/§3.4 让系统在 T3 替应聘方"默认放弃",
   §3.2 让递补在平台内自动发生——这些"替用户决策"提了效率,但可能伤
   体验。NPS 跌穿基线即代表效率是拿应聘方满意度换的。
2. **客服工单率不能涨。** §3.3 容忍"招聘方滥取消"的可能性(MVP 不分级
   退款),用观察代替治理。若取消相关工单率显著上涨,说明 §3.3 的"先
   观察"假设到期,该启动 v2 分级补偿,而不是继续容忍。
3. **中介方 #17 提交平均耗时不能涨过阈值。** §3.2 要求中介在 #17
   一次性排出 rank 1~N + N+1~ 备选——这是递补能自动化的代价。若 #17
   耗时涨到中介开始敷衍排序,递补质量会反噬北极星。

护栏指标只设"不得恶化"方向,不设增长目标。

## §4 证伪逻辑:什么结果出现,我就认为设计错了

> 这一节是本文档的核心,也是面试"你怎么验证这个设计是对的"的标准答法。
> 每条都给出:**触发条件 → 这推翻了哪条设计假设 → 下一步动作**。

- **§3.1 双重确认:** 若双段通过率 ≥ 75% **但**拉群后 24h 退群率并未
  低于 v2 基线 → 说明"应聘方没看到通知"这条失败并非主因,§3.1 拆出
  「待应聘方确认」的收益被高估;下一步不是加确认步骤,而是回到通知
  触达本身(推送到达率 / 打开率)。
- **§3.2 自动递补:** 若递补激活成功率 < 60% 且失败集中在"备选池耗尽"
  → 推翻"中介一次性排 N+M 个有序候选足够兜底"的假设;动作是调大
  #17 强制备选数下限,而非回退到 v2 的三方协作递补。
- **§3.3 统一取消:** 若招聘方主动取消率 ≥ 10% **且**取消时点集中在
  "应聘方已确认之后" → 推翻"MVP 阶段滥取消可容忍"的假设;这是 v2
  分级补偿从"延期项"转为"必做项"的量化扳机。
- **§3.4 不对称:** 若 T2 超时率显著高于 T1/T3 → 说明"招聘方对名单的
  判断不可由系统替代"在现实里代价过高;此时**不是**改成 T2 默认通过
  (那会破坏 §3.1 招聘方背书),而是缩短 T2 提醒间隔 / 增加 #18 决策
  辅助信息,先降无响应率再谈是否调整原则。

  应答率、AB 设计)属 **v2 · 本版不做**,定义在 `AI_MATCHING_SPEC.md`,
  不在本 MVP 指标体系内——本文档只衡量 §3.1–§3.4 这四条 v1 闭环决策,
  不把任何 AI 指标呈现为 MVP 已实现。
